# ADR-002: docs-mcp-server on homelab-2nd — local-only, SQLite on local-path, in-cluster Ollama embeddings

Date: 2026-06-26
Status: Accepted
Supersedes: nothing
Superseded by: nothing

## Context

The Supreme Leader wanted a self-hosted `docs-mcp-server` (arabold/docs-mcp-server) available only on the home LAN, so his M1 Max MacBook Pro is not burdened with running another service or an embedding model. The server provides an MCP/SSE endpoint plus a web UI for indexing documentation from websites, GitHub, local files, etc.

This raised three architectural questions:

1. **Ingress:** Should the service be exposed publicly via Cloudflare Tunnel, or kept local-only?
2. **Storage:** docs-mcp-server uses a SQLite database (`documents.db`) by default under `/data`. Should this be made durable via CNPG Postgres, or kept on `local-path`?
3. **Embeddings:** The server supports OpenAI, Ollama, Gemini, Azure, etc. For fully local operation, where does the embedding model run?

## Decision

**Deploy docs-mcp-server as a local-only service on `homelab-2nd`, using SQLite on a `local-path` PVC, with a co-located Ollama container serving `nomic-embed-text` for embeddings.**

### Details

- **Namespace:** `docs-mcp`.
- **Ingress:** No Cloudflare Tunnel, no Ingress, no cert-manager. LAN access via `hostPort: 6280` on the node (`192.168.1.179:6280`). A CoreDNS entry `docs-mcp.homelab-2nd.lan` points to the same IP for convenience inside the cluster; LAN clients may need a local DNS/hosts entry.
- **Storage:** Two `local-path` PVCs:
  - `docs-mcp-data` (50Gi) for the SQLite index and cached documents.
  - `docs-mcp-config` (1Gi) for runtime config.
  - `ollama-models` (20Gi) for the embedding model weights.
  All three are considered rebuildable cache/state. The index can be regenerated by re-scraping source docs; Ollama can re-pull `nomic-embed-text`.
- **Embeddings:** `ollama/ollama:latest` Deployment with an init container that pulls `nomic-embed-text` before the main container starts. docs-mcp-server connects to `http://ollama-embeddings.docs-mcp.svc.cluster.local:11434/v1` using a fake OpenAI API key (`ollama`) stored in a SOPS-encrypted secret for consistency with repo conventions.
- **Configuration:** A read-only ConfigMap provides `/config/docs-mcp-server/config.yaml` with `app.embeddingModel: openai:nomic-embed-text`, `telemetryEnabled: false`, `auth.enabled: false`, and `server.host: 0.0.0.0`.
- **Observability:** Logs are collected by the OpenTelemetry Collector DaemonSet; no Prometheus `/metrics` endpoint exists in either docs-mcp-server or Ollama, so no ServiceMonitor is configured. Pod resource metrics are available via kube-prometheus-stack cAdvisor/kubelet metrics as usual.

## Rationale

1. **Resource offload.** Moving docs-mcp-server and the embedding model to `homelab-2nd` keeps the M1 Max dedicated to running LLMs and Hermes.
2. **Local-only = simpler security model.** No public exposure means no OAuth2/OIDC, no Cloudflare Tunnel token, no TLS termination complexity. LAN trust is acceptable for this use case.
3. **SQLite is the default and only documented storage backend.** docs-mcp-server does not advertise Postgres support; forcing CNPG would require upstream changes or undocumented internals.
4. **Ollama is the path of least resistance for local embeddings.** `nomic-embed-text` is small (~274 MB), fast on CPU, and OpenAI-compatible.
5. **Architecture alignment.** Rebuildable cache/state belongs on `homelab-2nd` `local-path`; durable user/backup data belongs on OMV MinIO. This deployment fits the ephemeral bucket.

## Consequences

### Positive

- M1 Max CPU/GPU/RAM stay free for LLM inference and Hermes.
- Simple deployment: no CNPG cluster, no tunnel, no cert-manager.
- Fully local: documentation queries never leave the LAN.
- Fast local scraping/indexing on NVMe.

### Negative / Risks

- SQLite database is not durable. A node rebuild or PVC loss requires re-scraping all indexed documentation.
- `hostPort` ties the service to whichever node runs the pod (fine for single-node k3s, but brittle if we ever add workers).
- Ollama model weights are on `local-path`; re-pull is required after node rebuild.
- No native Prometheus metrics from docs-mcp-server or Ollama, so application-level observability is limited to logs and pod metrics.
- CoreDNS entry for `docs-mcp.homelab-2nd.lan` only helps cluster-side DNS; Mac clients still need a local DNS forwarder or `/etc/hosts` entry unless the home router handles `.homelab-2nd.lan`.

## Alternatives considered

| Option | Why rejected |
|---|---|
| Public Cloudflare Tunnel | Contradicts requirement for local-only access; adds token management and TLS complexity. |
| CNPG Postgres for docs-mcp | Not supported/documented by docs-mcp-server; would require upstream work. |
| Run Ollama on the M1 Max | Defeats the whole purpose of offloading from the Mac. |
| Use an external OpenAI embedding key | Violates the local-only goal and incurs cloud cost. |
| NodePort instead of hostPort | NodePort range is 30000-32767 by default; preserving port 6280 required hostPort. |

## When to revisit

Revisit this ADR if any of the following become true:

- docs-mcp-server gains official Postgres support and durable indexing becomes important.
- The SQLite index grows beyond 50Gi or the model cache beyond 20Gi.
- We add worker nodes to k3s and `hostPort` scheduling becomes problematic.
- Public access becomes desired (would require Cloudflare Tunnel + auth).
- We want long-term persistence of the index across node rebuilds (would require moving SQLite to an NFS share or adopting Postgres).
